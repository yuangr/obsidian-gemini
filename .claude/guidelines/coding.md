# Coding guidelines — Gemini Scribe

Repo-specific coding standards a senior reviewer enforces here. Generic quality principles
(DRY, SOLID, error handling, performance) come from the `code-review` skill; this file holds
what is **specific to this Obsidian plugin**. Architectural invariants live in `invariants.md`.

## Language, format, lint

- TypeScript-first. Group modules by domain; add barrel exports only when they simplify imports.
- **Prettier** (`npm run format`): 2-space indent, 120-column width, semicolons, single quotes,
  trailing commas. `.editorconfig` enforces LF endings and tabbed Markdown/config.
- **ESLint** (`npm run lint`, autofix `npm run lint:fix`) with the `eslint-plugin-obsidianmd`
  recommended preset — it enforces Obsidian-specific best practices (memory-leak prevention,
  cross-window safety, command-ID conventions, manifest validation). Pervasive rules pending a
  broader refactor are parked in `eslint.config.mjs` under `PERVASIVE_OBSIDIANMD_RULES_TODO`.
- camelCase for variables/functions, PascalCase for classes/types, kebab-case filenames aligned
  with their feature area.
- Handle TypeScript errors properly (all properties correctly typed); use proper `async`/`await`
  for all asynchronous operations.

## Logging — never `console` directly

The plugin uses a dedicated Logger service (`src/utils/logger.ts`) that respects the debug-mode
setting; it avoids global console patching so it doesn't conflict with other plugins.

- **Never** call `console.log()` / `console.debug()` directly.
- Access it as `this.plugin.logger` (components), `context.plugin.logger` (tool implementations
  via `ToolExecutionContext`), or accept a `logger` parameter (utility functions).
- `logger.log()` / `logger.debug()` output **only** when debug mode is on (auto-filtered — don't
  wrap them in a manual `if (settings.debugMode)` check). `logger.error()` / `logger.warn()` are
  **always** visible; use them for API errors, data-integrity issues, deprecations.
- Output is prefixed `[Gemini Scribe]`. Logger output stays English (it is not user-facing UI).

## User-facing strings go through i18n

- All static user-facing UI strings (settings, modals, agent view, command names, notices,
  tooltips) go through `t(key, vars?)` from `src/i18n/index.ts` — **never hardcode user-visible
  English**. `src/i18n/en.ts` (key → `{message, context}`) is the source of truth; per-language
  files are AI-generated by `npm run translate` and committed.
- **Stay English** (do NOT wrap in `t()`): model-facing strings (prompts, tool descriptions),
  strings persisted to vault files (history callout markers, frontmatter keys), logger output,
  and any string compared with `===`.

## Obsidian API first

Prefer built-in Obsidian API over low-level operations (see the `obsidian-plugin-development`
skill for depth):

- `vault.getMarkdownFiles()` over `vault.adapter.list()`
- `app.fileManager.processFrontMatter()` for frontmatter
- `vault.getAbstractFileByPath()` for file lookup; `app.metadataCache` for metadata
- `app.fileManager.renameFile()` for renames (preserves metadata/links)
- `app.workspace.openLinkText()` for clickable links in views
- Always use normalized paths and the metadata cache.

## Plugin type surface — never import `main.ts`

Reference the plugin only through the leaf interface:
`import type { ObsidianGemini } from '../types/plugin'` — **never** `from '../main'`. `main.ts`
imports nearly the whole codebase, so a type reference back to it re-introduces cycles (#1155).
New service handles on the plugin class must be mirrored in `src/types/plugin-services.ts`
(module augmentation). When a manager and a sibling need a shared type/helper, put it in a leaf
module both import — never import the manager back. See `invariants.md` for the acyclic-graph rule.
`ToolExecutionContext.plugin` is already typed `ObsidianGemini` — use `context.plugin` directly,
no cast.

## Platform guards

Desktop-only APIs (Electron, MCP, Node.js) must be guarded with Obsidian's `Platform` checks
(`Platform.isDesktop` / `isMobile` / `isIosApp` / `isAndroidApp`) — unguarded use crashes on
mobile. Regex lookbehind is iOS 16.4+ only; add fallbacks.

## UI / design tokens

Consume the theme-adaptive `--gs-*` design tokens at the top of `styles.css` — do **not** hardcode
colors or reach for raw `var(--obsidian-var)` / magic numbers. Icons are Lucide via `setIcon()`,
sized from `--gs-icon-*` (default `lg` = 18px). Full reference: `docs/contributing/design-system.md`.
For a value-preserving CSS/token change, verify no computed-value change in a live vault
(`obsidian eval` + `getComputedStyle`) rather than eyeballing.

## Don't hand-edit generated output

- Generated bundles `main.js`, and generated sources `src/services/generated-help-references.ts`
  (produced from `docs/guide/` + `docs/reference/` at build) — never hand-edit; change the source.
- i18n per-language files (`src/i18n/ru.ts`, …) are AI-generated — edit `en.ts` and regenerate.
- System-folder protection: always exclude the plugin state folder (`settings.historyFolder`) and
  `.obsidian` from vault file operations.
